The implementation must have a working `tailwindcss` CLI available without

**Path A — pre-provisioned binary (preferred):** Check whether `tailwindcss` is

`package.json`. Document the `npm install` step in the README.

**Never** attempt an ad hoc download (e.g. `curl | sh`, `wget` of a release
binary, or a one-off `npm install -g`) as the sole provisioning mechanism

is not reproducible.

### 2.3 Pre-paint theme script (Plan §7.3)

- [x] Implement the four-step precedence algorithm as an inline `<script>` in
  `base.html <head>`:
  1. If `formy.themeUserSet === "true"` and `formy.theme` is `"light"` or
     `"dark"`, apply `formy.theme` (user-set preference wins; ignore `?theme=`).
  2. Else if `?theme=` is exactly `"light"` or `"dark"`, apply it and write
     to `formy.theme` (GoLightly initial hint; do **not** write
     `formy.themeUserSet`).
  3. Else if `formy.theme` is exactly `"light"` or `"dark"`, apply it
     (previously stored hint).
  4. Else apply `"light"` (cold default).
- [x] Applying a theme means: toggle `dark` class on `document.documentElement`
  and set `document.documentElement.style.colorScheme` to the resolved value.
- [x] Any malformed `?theme=` value (anything other than `"light"` or `"dark"`)
  must be silently discarded — the page must never break.
- [x] The script must run **before** the body is painted to prevent a theme
  flash.

### 2.4 In-app toggle handler (Plan §7.4)

- [x] Add a theme toggle button to the sticky header in `base.html`.
- [x] Attach an inline `onclick` handler (or a small `<script>` block after the
  button) that:
  1. Reads the current theme from
     `document.documentElement.classList.contains('dark')`.
  2. Computes the next theme (`dark` → `light`, `light` → `dark`).
  3. Toggles the `dark` class and sets `style.colorScheme`.
  4. Writes `localStorage.setItem('formy.theme', nextTheme)`.
  5. Writes `localStorage.setItem('formy.themeUserSet', 'true')`.
- [x] The `formy.themeUserSet` write (step 5) is what distinguishes a user
  action from a GoLightly URL-parameter hint.

## Cross-phase constraints (always in effect)

- **No `shell=True`** in any `subprocess.run` call — all CLI invocations must
  use an explicit argument list. (Plan §10.3, §10.4.4.)
- **No secrets in command strings or logs** — Formy02 does not set `PGPASSWORD`
  because it uses passwordless local PostgreSQL trust auth; restore-only
  `PGOPTIONS` goes in the subprocess `env` dict. (Plan §10.6.)
- **No stack traces or DB errors in production responses** — all error handlers
  sanitize `details` in production. (Plan §9, ERROR_REQUIREMENTS.md.)
- **All `.md` files** created or modified carry the four-key YAML frontmatter
  per `AGENTS.md`. The implementing agent sets `created_by` and `modified_by`
  to its own identity (`<agent name> (<model>)`, lowercase, no email, no angle
  brackets). The specific agent name is not prescribed by this TODO; AGENTS.md
  is authoritative.
- **Commit messages** follow `AGENTS.md`: lowercase title ≤50 chars, ideal
  `type: message` format, body for non-trivial changes, `co-authored-by:` line
  with the implementing agent's identity.
- **Dark-mode safety** — every template surface (card, input, border, text,
  button) must use semantic token classes that resolve correctly under `.dark`.
  Never use hard-coded light-mode colors (`bg-white`, `text-calm-800`, etc.)
  on primary page building blocks. (Plan §6.)
- **ORM session isolation for restore** — the restore route must use
  `require_admin_for_restore` (not `require_admin`) and call
  `db_backup.restore_backup(dump_bytes)` with no ORM session open. (Plan §10.4.)
- **Audio name/DataTransfer contract** — hidden input `name="audio_<question_id>"`,
  DataTransfer mechanism, and `File` MIME type `audio/webm` must not change
  between phases. (Plan §8.1.1.)
- **Tailwind tooling** — use whichever tooling path (A, B, or C) was
  established in Phase 2.1 consistently across all phases. Never attempt an
  ad hoc network download of the Tailwind CLI during implementation.
- **Logout is POST** — any logout control in templates (header nav, standalone
  logout page) must submit `POST /logout` via a `<form method="post"
  action="/logout">` element. A plain `<a>` link to `/logout` would issue
  `GET /logout`, which is not a registered route and violates the preserved
  Formy auth contract.
- **Admin whitelist env var** — `EMAIL_ADMIN_USER` is the env var; never
  introduce `ALLOWED_ADMIN_EMAILS` as an env var. `config.ALLOWED_ADMIN_EMAILS`
  is a derived Python set only.
