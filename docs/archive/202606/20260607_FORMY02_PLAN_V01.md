# Formy02 — Implementation Plan V01

Planner: claude (opus-4.8). Source requirements:
[20260607_FORMY02_PRD.md](20260607_FORMY02_PRD.md). This is an architectural
overview of the technology, the general flow, and the key modules/functions — not
a task list. The task-style TODO comes after this plan clears assessment.

## 1. Strategy and scope

Formy02 is a re-skin + datastore swap of the existing Formy app, not a rewrite.
The plan is to **port Formy's Python backend into this repo essentially intact**,
then make four scoped changes on top of it:

1. Replace SQLite with PostgreSQL at the engine/config layer.
2. Replace the Bootstrap-based Jinja templates + `static/style.css` with a
   Tailwind build that reproduces GoLightly04's design tokens and components.
3. Add a theme system (light/dark) with a pre-paint hand-off from a `?theme=`
   URL parameter, plus an in-app toggle.
4. Give the public questionnaire a playful hero with a waving-photo placeholder.

Backend route signatures, models, auth, email, audio, logging, and error
handling are preserved in behavior; most edits are in templates, config, the DB
engine, and a small amount of new theme/static plumbing.

A fifth, independent change lives in the **GoLightly04 repo** (forward theme on
the footer link) and is described in §11. It is small and isolated; it could be
its own micro-plan, but it is documented here to keep the feature coherent.

## 2. Technology

- Language/runtime: Python 3.13 in a project-local `venv` (per `AGENTS.md`).
- Web: FastAPI + Jinja2 (`Jinja2Templates`), Uvicorn, server-rendered HTML.
- ORM: SQLModel / SQLAlchemy.
- Database: PostgreSQL via the `psycopg` (v3) driver.
- Styling: Tailwind CSS built to a static stylesheet served from
  `src/app/static/`. Recommended: the standalone Tailwind CLI so no Node app is
  introduced; a `tailwind.config.js` + input CSS hold the ported tokens. (Final
  tooling pick is the one open call-out for the assessor — see §13.)
- Auth: bcrypt password hashing, `python-jose` JWT, `itsdangerous` signed
  session cookies (all already in Formy's `requirements.txt`).
- Email: Gmail SMTP (existing `utilities/email.py`).
- Logging: Loguru per `docs/LOGGING_PYTHON_V07.md`.

Dependencies added vs. Formy: `psycopg[binary]` (Postgres driver). Dependency
removed: implicit SQLite usage. Tailwind is a build-time tool, not a Python dep.

## 4. Data layer (SQLite → PostgreSQL)

Reference: `Formy/src/app/models/database.py`, `on_start_up.py`.

- `models/database.py`: replace the SQLite `DATABASE_URL` and
  `connect_args={"check_same_thread": False}` with a Postgres URL assembled from
  config (`postgresql+psycopg://user:pass@host:port/dbname`). Keep
  `create_engine`, `create_db_and_tables()` (still `SQLModel.metadata.create_all`),
  and `get_session()` as the FastAPI dependency.
- Schema is unchanged in shape (PRD §8.2): `User`, `SurveyedPerson`, `Question`,
  `Response`, `ContractSurveyedPersonQuestionResponse`. One type concern:
  `Question.options` (list[str]) should map to a Postgres JSON/ARRAY column
  rather than the SQLite text encoding — confirm the SQLModel column type during
  implementation.
- Startup flow in `main.py` lifespan is preserved: `configure_logging()` →
  `check_database()` (creates tables) → `create_admin_user()` (seeds whitelisted
  admins from `ALLOWED_ADMIN_EMAILS`). For Postgres, `check_database()` should
  also surface a clear fatal error if the database is unreachable, consistent with
  the logging doc's fail-fast posture.
- Audio bytes remain on the filesystem under `AUDIO_FILES_PATH`; only paths are
  stored. No change to `utilities/audio.py:save_audio_file`.

## 5. Configuration and environment

Reference: `Formy/src/app/utilities/config.py`.

- Remove the SQLite keys `NAME_DB` / `PATH_DATABASE`. Add Postgres keys
  (host, port, db name, user, password, optional schema) loaded via the existing
  `_require` / `_require_for_env` helpers. Postgres connection vars should be
  required in all environments (a Postgres app cannot run without them).
- Rename logging size var to `LOG_MAX_SIZE_IN_MB` to match
  `docs/LOGGING_PYTHON_V07.md` (Formy used `LOG_MAX_SIZE`); keep `LOG_MAX_FILES`.
- Keep all auth/email keys, `URL_BASE_WEBSITE`, `AUDIO_FILES_PATH`,
  `PATH_PROJECT_RESOURCES`, `NAME_APP`, `RUN_ENVIRONMENT`.
- Update `.env.example` accordingly; document every key in the README.

## 6. Styling system (Tailwind port of GoLightly04)

Reference: `GoLightly04/web/tailwind.config.js`,
`GoLightly04/web/src/styles/globals.css`.

- `tailwind.config.js`: copy the `primary` / `secondary` / `accent` / `calm`
  palettes, the semantic `canvas`/`raised`/`inset`/`overlay`/`subtle`/`strong`/
  `ink`/`ink-muted` colors (driven by CSS variables), the `display`/`sans` font
  families, `maxWidth.app`, and the gentle keyframe/animation. Set
  `darkMode: 'class'`. Point `content` at `src/app/templates/**/*.html`.
- Input CSS (`src/styles/tailwind.css`): `@tailwind base/components/utilities`,
  the `:root` and `.dark` CSS-variable blocks from globals.css, and the
  `@layer components` classes `.btn-primary`, `.btn-secondary`, `.btn-outline`,
  `.input-field`, `.card`, `.section-heading`.
- Build step: Tailwind CLI compiles to `src/app/static/app.css`, referenced from
  `base.html`. Document the build command in the README and `package`/`Makefile`
  note. FastAPI's existing `/static` mount serves it unchanged.
- Templates: rewrite `base.html` to load the Tailwind stylesheet and define the
  GoLightly-style shell (sticky `backdrop-blur` header with brand + theme toggle,
  gradient page background, minimal footer). Convert each page template from
  Bootstrap classes to the Tailwind/component classes. Templates affected:
  `base.html`, `questionnaire.html`, `thank_you.html`, `login.html`,
  `register.html`, `forgot_password.html`, `reset_password.html`,
  `email_reset_password.html`, `admin/index.html`, `admin/questions.html`,
  `admin/responses.html`.

## 7. Theme system and hand-off

Reference: `GoLightly04/web/src/components/ThemeToggle.tsx` (behavior to mirror).

- Pre-paint script in `base.html` `<head>`: read `?theme=` (validate to exactly
  `light`/`dark`), else Formy02's stored preference (cookie/localStorage key,
  e.g. `formy.theme`), else default `light`; set/remove `dark` on
  `document.documentElement` and set `style.colorScheme` before body renders to
  avoid flash. Persist the resolved value.
- In-app toggle: a small inline script + button mirroring GoLightly's toggle
  (flip `.dark`, update `color-scheme`, write storage). Once the user toggles,
  the Formy02 preference overrides any future incoming param within the same
  browser.
- The param is an initial hint only; server code does not need to read it
  (theme is resolved client-side), which keeps routes untouched. This is a
  deliberate choice to minimize backend change — note for assessor in case a
  server-side/no-JS theme is preferred instead.

## 8. Page flows (behavior preserved, presentation restyled)

Reference: `Formy/src/app/routes/`.

- Public questionnaire (`GET /` → `questionnaire_page`): same query for
  `is_active` questions ordered by id; template gains a playful hero with the
  waving-photo placeholder (`static/images/waving_placeholder.png`) and warm,
  anonymity-reassuring copy. Question rendering (yes/no radios, multiple-choice
  radios, text area) and the per-question MediaRecorder audio widget are
  preserved, re-skinned to the new component classes.
- Submit (`POST /submit` → `submit_questionnaire`): unchanged logic — create
  `SurveyedPerson`, persist only non-blank/with-audio responses, link via
  junction, save audio with the zero-padded dated filename, redirect to
  `/thank-you`.
- Thank-you (`GET /thank-you`): restyled, with a friendly return path.
- Auth (`routes/auth.py`): register (whitelist check), send verification email
  to `{URL_BASE_WEBSITE}/verify?token=<jwt>`, `GET /verify`, login/logout,
  forgot/reset password — all preserved; only templates change.
- Admin Questions (`admin/questions.html`): searchable/sortable questions table
  + create form (Question, Type, Active, Options) restyled as GoLightly cards.
  Client-side search/sort behavior preserved.
- Admin Responses (`admin/responses.html`): table grouped by `SurveyedPerson`
  with the Questions-count column opening a modal of answers + audio playback,
  restyled (the modal adopts GoLightly's overlay/`backdrop-blur` treatment).

## 9. Logging and error handling

- `utilities/logging_setup.py`: align to `docs/LOGGING_PYTHON_V07.md` — single
  daily-or-size rotating file sink, env-driven sinks/levels, `enqueue` in
  testing/production, `sys.excepthook` for uncaught exceptions, early-exit logs,
  `LOG_MAX_SIZE_IN_MB` naming. Invoked first in the lifespan.
- `utilities/error_handlers.py`: keep the three handlers (HTTP, validation,
  unhandled). JSON error bodies follow `docs/ERROR_REQUIREMENTS.md`
  (`{ error: { code, message, details?, status } }`, sanitized in production);
  HTML routes render friendly themed error pages while logging full detail.

## 10. Backup utility

- Adapt `utilities/db_backup.py` from copying a SQLite file to a `pg_dump`-based
  dump (and a documented restore path). Output location under
  `PATH_PROJECT_RESOURCES`. Exact CLI invocation/scheduling is a TODO-phase
  detail; the plan commitment is a working backup + restore for Postgres.

## 11. GoLightly04-side change (separate repo)

Reference: `GoLightly04/web/src/components/AppShell.tsx` (footer "Help Us
Improve" anchor → `https://formy.go-lightly.love`), `ThemeToggle.tsx`
(`golightly.theme` in localStorage, `.dark` on `<html>`).

- Change the footer anchor so its `href` carries the current theme, computed at

- Isolated, low risk; testable independently. Recommend implementing it as the

## 13. Sequencing, risks, and assessor call-outs

Suggested build order: (1) port backend + Postgres engine/config; (2) Tailwind
build + `base.html` shell + theme system; (3) restyle questionnaire/thank-you +
waving hero; (4) restyle auth + admin pages; (5) logging/errors/backup alignment;
(6) docs; (7) GoLightly footer change.

Risks / decisions to vet:
- Tailwind tooling: standalone CLI vs. minimal Node/PostCSS. Recommended:
  standalone CLI to avoid introducing a Node project into a Python app.
- `Question.options` column type under Postgres (JSON vs. ARRAY).
- Client-side-only theme resolution (no-JS users get the default theme) — confirm
  acceptable, or add a server-read cookie fallback.
- `CREATE_DATABASE_DOCS.md` adaptation depth for a non-monorepo Python app.
- Whether any seed questions ship for the first deploy (PRD §15.5).
