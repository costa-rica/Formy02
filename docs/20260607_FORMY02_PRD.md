# Formy02 — Product Requirements Document

## 1. Summary

Formy02 is a redesigned version of the existing Formy research questionnaire app
(`/Users/nick/Documents/Formy`). It keeps Formy's full utility but restyles the
entire UI to match the look and feel of the GoLightly04 web app
(`/Users/nick/Documents/GoLightly04/web`), gives the public questionnaire page a
warm, playful vibe — including a photo of the site owner waving — and moves the
data store from SQLite to PostgreSQL.

The public questionnaire is intended as a low-friction feedback channel for people
the owner meets while canvassing. GoLightly04's site footer already links to it
("Help Us Improve" → `https://formy.go-lightly.love`), so Formy02 should feel like
a natural, friendly extension of the GoLightly brand to anyone who follows that
link.

## 2. Background and context

- Formy (the reference app) is a self-hosted FastAPI + Jinja2 + SQLite app.
  It supports a public questionnaire (text / yes-no / multiple-choice questions
  with optional in-browser audio recording), admin authentication with email
  verification, an admin Questions manager, and an admin Responses viewer.
- GoLightly04/web (the style reference) is a Next.js / React / TypeScript /
  Tailwind app with a calm, modern aesthetic: soft slate neutrals, sky-blue and
  purple accents, rounded cards, gentle shadows and gradients, a display
  typeface, and a class-based light/dark theme toggle.
- This project's `docs/` directory carries forward-looking conventions for
  logging, errors, API docs, database docs, planning, markdown frontmatter, and
  commit messages. Where these conflict with how the original Formy did things,
  the `docs/` conventions win (see §13 and §15).

## 3. Goals and non-goals

### 3.1 Goals

- Preserve 100% of Formy's existing utility (every feature in §6).
- Restyle the entire app to match GoLightly04's design system (§5).
- Make the public questionnaire page feel playful and welcoming, anchored by a
  photo of the owner waving (§6.1).
- Move persistence from SQLite to PostgreSQL (§8).
- Add a light/dark theme toggle and accept an initial theme handed off from
  GoLightly04 via a URL parameter (§7).
- Apply this project's `docs/` conventions for logging, errors, and
  documentation going forward (§10–§13).

### 3.2 Non-goals

- Rewriting the backend in Next.js/Node. The stack stays Python (FastAPI +
  Jinja2). Only the presentation layer and the data store change.
- Building a public-facing account system. Only admins authenticate; the
  questionnaire stays anonymous and open.
- Changing the questionnaire's data model semantics or the audio capture
  mechanism beyond what restyling and the Postgres move require.
- Redesigning GoLightly04 itself. The only change there is making one existing
  footer link forward the current theme (§7.3).

## 4. Users and use cases

- Canvassing contacts (primary, anonymous): a person the owner meets follows the
  GoLightly "Help Us Improve" link, lands on a friendly questions page, and
  optionally leaves written and/or spoken feedback. No login.
- Admin (the owner): registers (email must be whitelisted), verifies via email,
  logs in, manages questions, and reviews responses (including audio playback).

## 5. Design system requirements

Formy02 must visually match GoLightly04. Because Formy02 stays on Jinja2 (not
React), the design tokens and component patterns are ported into the Jinja
templates and a Tailwind-built stylesheet rather than reused as React components.

### 5.1 Tokens to port from GoLightly04

Source of truth: `GoLightly04/web/tailwind.config.js` and
`GoLightly04/web/src/styles/globals.css`.

- Color palettes: `primary` (sky blue), `secondary` (purple), `accent` (green),
  `calm` (slate 50–950).
- Semantic surface/text/border CSS variables for theming:
  `--surface-canvas`, `--surface-raised`, `--surface-inset`, `--surface-overlay`,
  `--border-subtle`, `--border-strong`, `--text-ink`, `--text-ink-muted`, with the
  light values on `:root` and dark overrides under `.dark` (mirror globals.css).
- Fonts: display = `Trebuchet MS` family; sans = `Avenir Next` family.
- Component classes to replicate: `.btn-primary`, `.btn-secondary`,
  `.btn-outline`, `.input-field`, `.card`, `.section-heading`.
- Visual motifs: rounded-2xl/3xl cards, `shadow-sm`/`shadow-md`, subtle
  `backdrop-blur` headers, soft top-to-bottom gradients on page backgrounds.

### 5.2 Tailwind integration approach

The original Formy ships a single hand-written `static/style.css` and uses
Bootstrap-style classes in templates. Formy02 replaces this with Tailwind:

- Use the standalone Tailwind CLI (no full Node app required) to build a static
  stylesheet that Jinja templates reference, OR a minimal Node/PostCSS build —
  the exact tooling is a plan-phase decision. The PRD requirement is only that
  the produced CSS reproduces GoLightly04's tokens and component classes above.
- Tailwind `content` globbing must cover the Jinja templates so classes are not
  purged.
- The build artifact lives under `src/app/static/` and is served by FastAPI's
  static mount, exactly as `style.css` is today.

### 5.3 Layout shell

- A top navigation/header bar styled like GoLightly's: branded, `backdrop-blur`,
  sticky, with the theme toggle. Admin-only nav links appear when an admin is
  authenticated.
- A footer consistent with GoLightly's tone (e.g. a "Contact" link). Keep it
  minimal.
- Responsive: must look correct on mobile (canvassing contacts will mostly open
  it on phones) and desktop.

## 6. Functional requirements (carry over all Formy utility)

All features below already exist in Formy and MUST be preserved. The reference
implementations live under `/Users/nick/Documents/Formy/src/app/`.

### 6.1 Public questionnaire page (`GET /`) — playful redesign

- Render all `is_active` questions in source order as a single form.
- Optional contact fields: name, email, phone (a `SurveyedPerson`).
- Question types: `yes_no` (radio), `multiple_choice` (radio over options),
  `text` (textarea). Preserve the existing type handling.
- Per-question optional audio recording when `allows_for_audio` is true, using
  the MediaRecorder browser flow already in Formy (record → stop → playback →
  attach to hidden file input → upload as multipart). Re-skin the controls but
  keep the mechanism.
- All questions optional. Blank or whitespace-only answers are not saved.
- Playful / welcoming treatment (new):
  - A friendly hero area at the top of the page featuring a photo of the owner
    waving. For this phase, use a clearly-labeled placeholder image (e.g.
    `src/app/static/images/waving_placeholder.png`); the real photo is dropped in
    during implementation. The layout must not break when the final image (a
    different aspect ratio) is swapped in.
    - The PRD permits a small playful flourish (e.g. a gentle wave animation or a
      warm greeting headline). Final placement is the implementer's call within
      a "warm, playful, on-brand with GoLightly" guideline.
  - Welcoming, plain-language copy that frames this as quick, optional feedback
    and reassures that it's anonymous unless they choose to share contact info.
- Submit posts to `POST /submit`.

### 6.2 Submission (`POST /submit`)

- Create a `SurveyedPerson` from the contact fields (all optional).
- For each active question, save a `Response` only when there is non-blank text
  and/or an audio upload; link person↔question↔response via the junction table.
- Save audio files to the filesystem under `AUDIO_FILES_PATH` using the existing
  dated-subdirectory + zero-padded naming convention
  (`SurveyedPerson_0001_Question_001_YYYYMMDD_HHMMSS.<ext>`); store the full path
  in the response row. Audio bytes stay on disk, not in Postgres.
- Redirect to the thank-you page on success.

### 6.3 Thank-you page (`GET /thank-you`)

- Restyled confirmation page with the same warm tone and a path back to GoLightly
  / the questionnaire.

### 6.4 Admin authentication

Preserve the full flow from Formy's `routes/auth.py` and `utilities/auth.py`:

- Register (`/register`): only emails in the `EMAIL_ADMIN_USER` whitelist may
  register; creates a user with `email_verified = false`.
- Email verification: send a verification email containing a link
  `{URL_BASE_WEBSITE}/verify?token=<jwt>` (30-minute expiry, carries user id +
  email); `GET /verify` flips `email_verified` to true.
- Login / logout with signed session cookie / JWT as currently implemented.
- Forgot password / reset password flow and the reset email template.
- Unverified users cannot log in.

### 6.5 Admin Questions page

- Admin-only. Two sections (restyled to GoLightly cards):
  - Questions table with columns Question ID, Question, Type, Active; searchable
    and sortable.
  - Create-question form: Question, Type, Active, Options (multiple choice only).
- Create / update / delete questions.

### 6.6 Admin Responses page

- Admin-only. Table of responses grouped by `SurveyedPerson` with columns:
  Surveyed Person ID, Name, Email, Phone, Questions (count of non-null
  responses), Updated At.
- The Questions count opens a modal listing that person's answered questions and
  responses, with a play button for any audio file. Modal shows the Updated At at
  top.

### 6.7 Database backup utility

- Preserve a backup capability equivalent to Formy's `utilities/db_backup.py`,
  adapted for PostgreSQL (e.g. `pg_dump`-based) rather than copying a SQLite
  file. Exact mechanism is a plan-phase detail; the requirement is that an
  operator can produce and restore a backup.

## 7. Theme handoff from GoLightly04

### 7.1 Goal

When a visitor clicks "Help Us Improve" in GoLightly04, Formy02 should open in the
same theme (light or dark) the visitor was using on GoLightly04, so the
experience feels continuous across the two sites.

### 7.2 Formy02 side

- Accept an initial theme via query parameter, e.g. `?theme=dark` or
  `?theme=light`, on the questionnaire page (and tolerate it app-wide).
- Apply the theme before first paint to avoid a flash of the wrong theme: an
  inline head script reads the `theme` param (falling back to Formy02's own stored
  preference, then to a default) and sets the `dark` class on `<html>`.
- Persist the resulting choice in Formy02's own storage (localStorage and/or
  cookie) so it sticks on subsequent visits and navigations.
- Provide Formy02's own theme toggle (mirroring GoLightly's `ThemeToggle`
  behavior: toggles the `.dark` class and `color-scheme`, persists the choice).
  Once the user toggles in Formy02, their Formy02 preference takes precedence over
  the incoming param.
- Treat the incoming param as an initial hint only; never let a malformed value
  break the page (validate to exactly `light`/`dark`).

### 7.3 GoLightly04 side (small change)

The only change to GoLightly04 is to make the existing footer link forward the
current theme. Reference:

- The link already exists:
  `GoLightly04/web/src/components/AppShell.tsx` (the "Help Us Improve" anchor to
  `https://formy.go-lightly.love`).
- GoLightly stores its theme in `localStorage` under `golightly.theme` and
  reflects it as the `.dark` class on `<html>` (see
  `GoLightly04/web/src/components/ThemeToggle.tsx`).
- Change: compute the link's `href` to include the current theme at click time
  (or reactively), e.g. read `document.documentElement.classList.contains('dark')`
  / the `golightly.theme` value and append `?theme=dark` or `?theme=light`.

### 7.4 Feasibility assessment

This is a small, low-risk change, for two reasons: the footer link already exists
and already points at Formy02's domain, and a URL parameter is exactly the right
transport because the two sites are different origins (localStorage cannot be
shared across `go-lightly.love` and `formy.go-lightly.love`). The only nuances —
both minor — are computing the href from the live theme at click time and
preventing a theme flash on Formy02 with a pre-paint inline script. No pushback;
recommended as scoped.

## 8. Data and persistence

### 8.1 Engine

- PostgreSQL replaces SQLite. Continue using SQLModel / SQLAlchemy as the ORM
  layer (keeps the Python model definitions close to Formy's) with a Postgres
  driver. Connection configured via environment variables.

### 8.2 Schema (carried over from Formy, unchanged in shape)

- `Users` — id, username, password (hashed), email, is_admin, email_verified,
  created_at, updated_at.
- `SurveyedPerson` — id, name?, email?, phone?, created_at, updated_at.
- `Questions` — id, question, is_active, type (`yes_no` | `text` |
  `multiple_choice`), allows_for_audio (default true), options (list, multiple
  choice only), created_at, updated_at.
- `Responses` — id, response?, audio_file_path_and_filename?, summary?,
  created_at, updated_at.
- `ContractSurveyedPersonQuestionResponse` (junction) — id, surveyed_person_id,
  question_id, response_id, created_at, updated_at.

### 8.3 Migration notes

- This is a fresh build, not a live data migration; no SQLite data needs to be
  carried over (confirm in plan phase if any seed data is wanted).
- `options` (list of strings) should map to a Postgres-appropriate column type
  (e.g. JSON/array) rather than the SQLite text encoding.
- Audio files remain on the filesystem; only their paths live in the database.

## 9. Environment configuration

Carry forward Formy's env surface, adjusted for Postgres and this project's
logging doc. Final list is a plan-phase detail; expected keys include:

- App: `NAME_APP`, `RUN_ENVIRONMENT` (`development` | `testing` | `production`),
  `URL_BASE_WEBSITE`, `PATH_PROJECT_RESOURCES`, `AUDIO_FILES_PATH`.
- Database (Postgres): host, port, database name, user, password, schema (replace
  Formy's `NAME_DB` / `PATH_DATABASE` SQLite keys).
- Auth/email: `EMAIL_ADMIN_USER` (whitelist), `SECRET_KEY`, `GMAIL_SMTP_USER`,
  `GMAIL_SMTP_APP_PASSWORD`, `GMAIL_SMTP_HOST`, `GMAIL_SMTP_PORT`.
- Logging: `PATH_TO_LOGS`, `LOG_MAX_SIZE_IN_MB`, `LOG_MAX_FILES` (align names with
  `docs/LOGGING_PYTHON_V07.md`).
- Required-variable startup validation must fail fast per the logging doc.

## 10. Logging

- Follow `docs/LOGGING_PYTHON_V07.md` (Loguru, `RUN_ENVIRONMENT`-driven sinks,
  single daily-or-size rotation file sink, `enqueue` in testing/production,
  `sys.excepthook` for uncaught exceptions, early-exit logging, fatal errors on
  missing required env vars). This supersedes Formy's older V05 logging doc.

## 11. Error handling

- Follow `docs/ERROR_REQUIREMENTS.md`: any JSON/API error responses use the
  standard `{ error: { code, message, details?, status } }` shape with
  machine-readable codes, sanitized in production. For server-rendered pages,
  show friendly themed error states and log full detail server-side.

## 12. Non-functional requirements

- Security: hashed passwords, signed cookies/JWT, admin pages gated, whitelist on
  registration, no secrets or stack traces leaked to clients in production.
  Because the questionnaire is publicly linked, treat it as exposed to the open
  internet (basic abuse resistance / input validation on submit).
- Accessibility: labeled inputs, sufficient contrast in both themes, keyboard
  operability of the audio controls and modal.
- Responsiveness: mobile-first; the questionnaire must be comfortable on a phone.
- Performance: the public page should load fast (small CSS bundle, optimized
  hero image).

## 13. Documentation and repo conventions

- Markdown in `docs/`: filenames `YYYYMMDD_DESCRIPTIVE_NAME.md` (caps,
  underscores); every file carries the four-key YAML frontmatter
  (`created_at`, `updated_at`, `created_by`, `modified_by`) per `AGENTS.md`.
- Commit messages follow `AGENTS.md` (lowercase title ≤50 chars, typed ideal
  format, body for non-trivial changes, `co-authored-by: <agent> (<model>)`).
- API docs: when an HTTP API surface exists, generate `docs/api-documentation/`
  per `docs/CREATE_API_DOCS.md`.
- Database docs: see conflict note in §15 — `docs/CREATE_DATABASE_DOCS.md` is
  written for a TypeScript/Sequelize `db-models` monorepo and does not directly
  fit a single Python/SQLAlchemy app; adapt its spirit (a `TABLE_REFERENCE` plus
  setup/usage guides) rather than following it literally.

## 14. Acceptance criteria

- Every Formy feature in §6 works in Formy02 against PostgreSQL.
- The public questionnaire, thank-you, login/register/verify/forgot/reset, admin
  Questions, and admin Responses pages all render in the GoLightly04 visual
  language in both light and dark themes.
- The questionnaire page shows the waving photo (placeholder acceptable for this
  phase) and reads as warm/playful.
- Audio record → playback → upload → admin playback works end to end.
- Opening Formy02 with `?theme=dark` (and `?theme=light`) applies that theme with
  no flash; toggling in Formy02 persists and overrides.
- The GoLightly04 footer "Help Us Improve" link forwards the current theme.
- Logging and error handling conform to the `docs/` standards; required env vars
  fail fast when missing.

## 15. Open questions and conflicts to resolve in the plan phase

1. Tailwind build tooling: standalone Tailwind CLI vs. a minimal Node/PostCSS
   build alongside the Python app — pick the lightest option that reproduces
   GoLightly's tokens.
2. `CREATE_DATABASE_DOCS.md` conflict: it assumes TS/Sequelize and a monorepo
   `db-models` package; decide how much to adapt for a Python/SQLAlchemy single
   app (recommended: adapt, don't follow literally).
3. Backup mechanism for Postgres (`pg_dump`/restore) and where backups live.
4. Final waving-photo placement, dimensions, and any animation flourish.
5. Whether any seed questions/data should ship for the first deploy.
6. Confirm the production domain (`formy.go-lightly.love`) and deployment target
   (Formy's reference includes an Ubuntu service runbook —
   `Formy/docs/UBUNTU_RUN_SERVICE.md` — to mirror).
