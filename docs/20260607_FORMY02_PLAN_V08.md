# Formy02 — Implementation Plan V08

Planner: claude (opus-4.8). Source requirements:
[20260607_FORMY02_PRD.md](20260607_FORMY02_PRD.md). V08 supersedes
[20260607_FORMY02_PLAN_V07.md](20260607_FORMY02_PLAN_V07.md), incorporating
the single concern raised in
[20260607_FORMY02_PLAN_V07_ASSESSMENT_CODEX.md](20260607_FORMY02_PLAN_V07_ASSESSMENT_CODEX.md):
V07's §10 restore route preserves Formy's `require_admin` auth dependency, which
yields a SQLModel session that remains open — holding an uncommitted transaction
on the app's Postgres connection — for the full duration of the request. Because
`pg_restore --clean` must acquire exclusive locks to drop and recreate tables
(including `users`, which the auth query touched), and because the app's own ORM
connection holds those locks while the request is blocked waiting for `pg_restore`
to return, the restore operation can deadlock. V08 revises §10 to require that the
restore route uses a restore-specific auth boundary that closes all ORM sessions
before `pg_restore` runs, mandates a bounded subprocess timeout, and explains why
the optional `pg_terminate_backend` call noted in V07 is not sufficient to resolve
the deadlock on its own. All other V07 sections are retained without material
change.

## 1. Strategy and scope

Formy02 is a re-skin + datastore swap of the existing Formy app, not a rewrite.
The plan is to **port Formy's Python backend into this repo essentially intact**,
then make four scoped changes on top of it:

1. Replace SQLite with PostgreSQL at the engine/config layer.
2. Replace the Bootstrap-based Jinja templates + `static/style.css` with a
   Tailwind build that reproduces GoLightly04's design tokens and components.
3. Add a theme system (light/dark) with a pre-paint hand-off from a `?theme=`
   URL parameter, plus an in-app toggle, governed by the precedence algorithm
   in §7.
4. Give the public questionnaire a playful hero with a waving-photo placeholder.

Backend route signatures, models, auth, email, audio, logging, and error
handling are preserved in behavior; most edits are in templates, config, the DB
engine, and a small amount of new theme/static plumbing. Three audio defects
present in the Formy reference are corrected (§8.1); the audio mechanism itself
is otherwise unchanged.

A fifth, independent change lives in the **GoLightly04 repo** (forward theme on
the footer link) and is described in §11. It is small and isolated.

## 2. Technology

- Language/runtime: Python in `/home/limited_user/environments/formy02` (per `AGENTS.md`).
- Web: FastAPI + Jinja2 (`Jinja2Templates`), Uvicorn, server-rendered HTML.
- ORM: SQLModel / SQLAlchemy.
- Database: PostgreSQL via the `psycopg` (v3) driver.
- Styling: Tailwind CSS built to a static stylesheet served from
  `src/app/static/`. Recommended: the standalone Tailwind CLI so no Node app is
  introduced; a `tailwind.config.js` + input CSS hold the ported tokens.
- Auth: bcrypt password hashing, `python-jose` JWT, `itsdangerous` signed
  session cookies (all already in Formy's `requirements.txt`).
- Email: Gmail SMTP (existing `utilities/email.py`).
- Logging: Loguru per `docs/LOGGING_PYTHON_V07.md`.
- Backup: `pg_dump` / `pg_restore` invoked via `subprocess.run` with an
  argument list (no `shell=True`).

Dependencies added vs. Formy: `psycopg[binary]` (Postgres driver). Dependency
removed: implicit SQLite usage. Tailwind is a build-time tool, not a Python dep.

## 4. Data layer (SQLite → PostgreSQL)

Reference: `Formy/src/app/models/database.py`, `on_start_up.py`.

### 4.1 Engine and session

`models/database.py`: replace the SQLite `DATABASE_URL` and
`connect_args={"check_same_thread": False}` with a Postgres URL assembled from
config (`postgresql+psycopg://user:pass@host:port/dbname`). Keep `create_engine`,
`create_db_and_tables()` (still `SQLModel.metadata.create_all`), and
`get_session()` as the FastAPI dependency.

### 4.2 Question.options — concrete JSONB mapping

**The problem fixed in V02:** Formy's `Question` model stores options via a
`options_json: Optional[str]` field backed by a `Column("options", Text)`, plus
a Python `@property` / `@setter` pair that calls `json.dumps` / `json.loads`.
V01 said "map to JSON/ARRAY" without naming a design; the Codex assessment
correctly identified that an implementer could leave the SQLite Text encoding in
place or accidentally collide with the existing property.

**The concrete mapping:** Replace the `Text`-backed `options_json` field and the
`@property` / `@setter` pair with a single mapped field using
`sqlalchemy.dialects.postgresql.JSONB`:

```python
from sqlalchemy.dialects.postgresql import JSONB

class Question(SQLModel, table=True):
    ...
    options: Optional[List[str]] = Field(
        default=None,
        sa_column=Column("options", JSONB),
    )
```

The DB column name stays `"options"` — the same name Formy uses — so there is
no migration rename. The `JSONB` type instructs SQLAlchemy to handle
serialization natively; no `json.dumps` / `json.loads` call is needed in
application code. The backing private field `options_json` and its property pair
are removed.

**Route and template contracts preserved without modification:**

- `routes/admin.py` create question: `q.options = parsed_options`
  — assigns a `list[str] | None` to the mapped field; works as-is.
- `routes/admin.py` update question: `q.options = [...]`
  — same assignment; works as-is.
- `templates/questionnaire.html` multiple-choice rendering: iterates `q.options`
  — receives a `list[str]` from the ORM; works as-is.
- `routes/admin.py` update question JSON path: reads
  `body["options"]` and assigns to `q.options`; works as-is because the ORM
  field accepts a Python list directly.

No changes to admin.py or questionnaire.html are required for options handling.
The only change is in `models/question.py`: swap the column type, drop the
private field, and remove the property/setter.

### 4.3 Startup flow — admin seeding removed

**The problem fixed in V02:** Formy's `on_start_up.py:create_admin_user()`
creates missing whitelisted users with `password=hash_password("test")`,
`is_admin=True`, and `email_verified=True`, and also auto-verifies any existing
unverified admin. If carried forward, Formy02 could launch with automatically
verified admin accounts protected by a known password, bypassing the very
registration and verification flow the PRD requires.

**The fix:** Remove `create_admin_user()` entirely. `on_start_up.py` contains
only `check_database()`, which calls `create_db_and_tables()`. The lifespan in
`main.py` calls only `configure_logging()` and `check_database()`.

The `ALLOWED_ADMIN_EMAILS` environment variable is retained — it continues to
gate registration in `routes/auth.py` (the whitelist check that returns 403 for
non-whitelisted addresses). It no longer drives any auto-seeding.

**First-deploy bootstrap:** An admin registers at `/register` with a
whitelisted email address, receives the verification email (requires SMTP to be
configured), clicks the link at `{URL_BASE_WEBSITE}/verify?token=<jwt>`, and
can then log in. This is the correct flow the PRD describes and the only
supported path.

If SMTP is unavailable during initial setup (a known operational edge case), the
operator must configure SMTP before first admin registration — not work around
it with startup seeding. The README documents this constraint explicitly.

### 4.4 Remaining schema

Schema shape is unchanged (PRD §8.2): `User`, `SurveyedPerson`, `Question`,
`Response`, `ContractSurveyedPersonQuestionResponse`. This is a fresh build
with no SQLite data to migrate; `create_db_and_tables()` on a clean Postgres
database creates all tables. Audio bytes remain on the filesystem under
`AUDIO_FILES_PATH`; only paths are stored.

### 4.5 Startup health check

For Postgres, `check_database()` should issue a lightweight connectivity check
(e.g. `SELECT 1`) before `create_all`, and log a fatal error and raise if the
database is unreachable — consistent with the `docs/LOGGING_PYTHON_V07.md`
fail-fast posture. An unreachable database must not silently allow the app to
start and serve 500s.

- Input CSS (`src/styles/tailwind.css`): include `@tailwind base/components/utilities`

  `@layer components` classes `.btn-primary`, `.btn-secondary`, `.btn-outline`,
  `.input-field`, `.card`, `.section-heading` — **but redefine each one around
  the semantic tokens, not around hard-coded light values.**

  **Critical constraint — do not copy GoLightly04's light-only helper class
  definitions verbatim when they are used as Formy02 primary page building
  blocks.** In `GoLightly04/web/src/styles/globals.css`, several helpers are
  defined with fixed light-mode values: `.card` uses `bg-white` with a fixed
  `border-calm-200` and no `.dark` variant; `.input-field` uses fixed light
  borders; `.section-heading` uses `text-calm-800`. GoLightly04 itself achieves
  dark-mode adaptation on actual page surfaces by using semantic utility classes
  directly — `bg-raised`, `border-subtle`, `text-ink`, `bg-inset`, `bg-overlay`
  — which are backed by the light/dark CSS variables declared in the `:root` /
  `.dark` blocks. Copying those helper definitions verbatim into Formy02 will
  produce white cards, hard light borders, and light-only text inside the dark
  theme, violating the PRD acceptance criterion that all pages render in
  GoLightly04's visual language in both light and dark themes.

  Required definitions for Formy02 component classes (use semantic tokens
  throughout):

  - `.card`: `bg-raised border border-subtle rounded-2xl shadow-sm` (use
    `shadow-md` for elevated cards). Do **not** use `bg-white`.
  - `.input-field`: `bg-inset border border-subtle text-ink placeholder:text-ink-muted
    rounded-lg focus:outline-none focus:ring-2 focus:ring-primary-500
    focus:border-primary-500`. The focus ring must use a semantic or palette
    color that is visible in both themes (e.g. `ring-primary-500`); do not use
    fixed light-only border or ring colors.
  - `.btn-outline`: `border border-strong text-ink hover:bg-raised rounded-lg
    px-4 py-2 transition-colors`. Border and text resolve via CSS variables and
    are legible in both themes.
  - `.section-heading`: `text-ink font-display` rather than a fixed
    `text-calm-800`.
  - `.btn-primary` and `.btn-secondary`: these reference palette colors
    (`primary-*`, `secondary-*`) that do not shift by theme and may be structured
    analogously to GoLightly04's definitions, but verify that the background and
    text colors produce sufficient contrast against both light and dark surface
    tokens.

  When the correct dark-mode behavior is achieved more clearly by applying
  semantic utility classes directly in Jinja templates rather than through a
  named component class, prefer the direct utility approach — the goal is correct
  dark-mode rendering in both themes, not strict adherence to the component-class
  abstraction.

- Build step: Tailwind CLI compiles to `src/app/static/app.css`, referenced from

- Templates: rewrite `base.html` to load the Tailwind stylesheet and define the

## 7. Theme system and hand-off

Reference: `GoLightly04/web/src/components/ThemeToggle.tsx` (behavior to
mirror), `GoLightly04/web/src/components/AppShell.tsx` (footer link to modify).

### 7.1 The core design requirement

The PRD (§7.2) states two rules that must hold simultaneously:

- The `?theme=` URL parameter is **an initial hint only** — GoLightly hands off
  the theme it is currently displaying so the visitor's experience feels
  continuous when they follow the footer link.
- Once the user **toggles in Formy02**, their Formy02 preference takes
  precedence over the incoming parameter.

V02's algorithm (check `?theme=` first, then fall back to `formy.theme`) violates
the second rule: any subsequent navigation to Formy02 from GoLightly with
`?theme=dark` or `?theme=light` would override a preference the user already
expressed by toggling. V03 fixes this with a two-key localStorage design.

### 7.2 Storage keys

Two `localStorage` keys govern theme state in Formy02:

| Key | Type | Meaning |
|-----|------|---------|
| `formy.theme` | `"light"` \| `"dark"` | The theme currently applied in this browser. Written by the pre-paint script (when accepting a GoLightly hint) and by the in-app toggle. |
| `formy.themeUserSet` | `"true"` | Set **only** when the user explicitly toggles via the Formy02 in-app button. Absent (or any other value) means no explicit local choice has been recorded. |

The `formy.themeUserSet` flag is the critical invariant: its presence means the
user's voice wins over any incoming URL parameter.

### 7.3 Pre-paint precedence algorithm

An inline `<script>` block in `base.html <head>` runs synchronously before the
body renders, preventing a theme flash. The algorithm in execution order:

1. **User-set preference wins.** If `formy.themeUserSet === "true"` and
   `formy.theme` is exactly `"light"` or `"dark"`, apply `formy.theme` and
   ignore `?theme=` entirely.
2. **GoLightly initial hint.** Else if `?theme=` is exactly `"light"` or
   `"dark"`, apply that value and write it to `formy.theme` (so subsequent
   navigation without a param stays consistent). Do **not** write
   `formy.themeUserSet`; the GoLightly hint is not a user-set preference.
3. **Previously stored hint.** Else if `formy.theme` is exactly `"light"` or
   `"dark"`, apply it.
4. **Cold default.** Apply `"light"`.

Applying a theme means: toggle the `dark` class on `document.documentElement`
and set `document.documentElement.style.colorScheme` to the resolved value.
Any malformed `?theme=` value (anything other than `"light"` or `"dark"`) is
silently discarded and the algorithm falls through to step 3 or 4 — the page
never breaks.

Pseudocode rendering of the pre-paint script:

```javascript
(function () {
  var stored = localStorage.getItem('formy.theme');
  var userSet = localStorage.getItem('formy.themeUserSet') === 'true';
  var param = new URLSearchParams(location.search).get('theme');
  var valid = function (v) { return v === 'light' || v === 'dark'; };

  var theme;
  if (userSet && valid(stored)) {
    // Step 1 — user explicitly toggled in Formy02; ignore incoming param
    theme = stored;
  } else if (valid(param)) {
    // Step 2 — GoLightly initial hint
    theme = param;
    localStorage.setItem('formy.theme', theme);
  } else if (valid(stored)) {
    // Step 3 — previously stored hint
    theme = stored;
  } else {
    // Step 4 — cold default
    theme = 'light';
  }

  document.documentElement.classList.toggle('dark', theme === 'dark');
  document.documentElement.style.colorScheme = theme;
}());
```

### 7.4 In-app toggle

The theme toggle button in `base.html` calls a small inline handler on click.
The handler:

1. Reads the current theme from `document.documentElement.classList.contains('dark')`.
2. Computes the next theme (`dark` → `light`, `light` → `dark`).
3. Toggles the `dark` class and sets `style.colorScheme`.
4. Writes `formy.theme = <next>` to `localStorage`.
5. Writes `formy.themeUserSet = "true"` to `localStorage`.

Step 5 is what distinguishes a user action from a GoLightly hint. Once
`formy.themeUserSet` is `"true"`, every future page load in this browser takes
step 1 of the pre-paint algorithm and the `?theme=` parameter is ignored.

The toggle mirrors GoLightly's `ThemeToggle.tsx` behavior (flip `.dark`, update
`color-scheme`, persist to storage) while adding the `formy.themeUserSet` write.

### 7.5 Server-side role

The server does not need to read `?theme=` or any theme cookie. Theme resolution
is entirely client-side: the pre-paint inline script runs before CSS is applied,
eliminating the need for a server-set class. No route changes are required for
theme support.

The `?theme=` parameter flows harmlessly through all routes (FastAPI ignores
unrecognized query params) — no server filtering or validation is needed.

### 7.6 No-JS behavior

Visitors with JavaScript disabled receive the default light theme. This is
acceptable for this deployment; the admin audience uses the app authenticated
and in a controlled browser environment. A server-read cookie fallback is noted
as a future enhancement but is not required for the first implementation (see
§13 open decisions).

### 8.1 Audio subsystem — three required fixes

The Formy reference contains three concrete audio defects. V05 instructed
implementers to "preserve" the audio mechanism, which is correct for the
overall flow (MediaRecorder → Blob → DataTransfer → hidden `<input>` →
multipart upload → filesystem save → admin playback). However, those three
defects must not be copied; they would leave the PRD acceptance criterion —
"Audio record → playback → upload → admin playback works end to end" —
unmet. Each defect and its fix is specified below.

#### 8.1.1 Questionnaire recorder — initial visibility

**The defect:** `questionnaire.html` renders the Record button with a Bootstrap
`d-none` class (equivalent to `display: none`). The JS initialization code
only hides the Record button after it is clicked and reveals it after Stop; no
code ever removes `d-none` on page load. As written, users can never start a
recording.

**The fix:** In `questionnaire.html`, render the Record button **visible by
default** for each question where `allows_for_audio` is true. The Stop button
and the playback `<audio>` element remain hidden on initial render — they are
only revealed by the JS event handlers after a recording is made, exactly as
before. The JS handlers themselves need no changes to their hide/show logic; the
only change is the HTML initial state.

Additionally, add a one-time JS check on `DOMContentLoaded` (or at the top of
the existing IIFE) that tests `typeof MediaRecorder !== 'undefined'` and, if
`MediaRecorder` is unavailable, hides the Record button and sets the status text
to a graceful-degradation message. This ensures the widget degrades cleanly in
browsers that do not support the API.

**Preserved invariants:**

- The hidden `<input type="file">` element's `name` attribute remains
  `audio_<question_id>` (e.g. `name="audio_1"`). The route handler at
  `POST /submit` reads fields by this name to associate audio with the
  correct question.
- The DataTransfer mechanism — constructing a `DataTransfer`, adding a `File`,
  and assigning its `files` to the hidden input — is unchanged. This is the
  correct cross-browser approach for attaching a Blob to a file input before
  multipart form submission.
- The `File` name in the DataTransfer is `audio_<qid>.webm` and the MIME type
  is `audio/webm`; these become the upload's filename and content-type as seen
  by the server, which is the correct input for §8.1.2.

#### 8.1.2 Audio file saving — extension derived from upload

**The defect:** `utilities/audio.py` hardcodes the `.mp3` extension in the
filename regardless of the actual file the browser uploads. The browser
MediaRecorder produces a `audio/webm` Blob named `audio_<qid>.webm`. The saved
file therefore has a `.mp3` extension but WebM content, which causes playback
failures in browsers that rely on extension-based MIME sniffing and makes the
admin MIME type (§8.1.3) impossible to derive correctly.

**The fix:** In `utilities/audio.py`, derive the file extension from the
`UploadFile` before building the destination filename:

1. Take `Path(upload.filename).suffix` if `upload.filename` is non-empty and
   the suffix is a known audio extension (`.webm`, `.ogg`, `.mp4`, `.m4a`,
   `.mp3`, `.wav`, `.aac`). This is the primary signal; the browser always sets
   the filename to `audio_<qid>.webm`.
2. If step 1 yields no recognized extension, fall back to mapping
   `upload.content_type` to an extension:
   - `audio/webm` → `.webm`
   - `audio/ogg` → `.ogg`
   - `audio/mp4` → `.m4a`
   - `audio/mpeg` → `.mp3`
   - `audio/wav` → `.wav`
   - `audio/aac` → `.aac`
3. If neither step produces a result, fall back to `.webm` as the safe default
   (it is what the MediaRecorder produces in all modern browsers when no explicit
   MIME type is negotiated).

**Preserved invariants:**

- The dated subdirectory structure (`{AUDIO_FILES_PATH}/YYYYMMDD/`) is
  unchanged.
- The zero-padded filename stem
  (`SurveyedPerson_{id:04d}_Question_{qid:03d}_YYYYMMDD_HHMMSS`) is
  unchanged. Only the extension changes.
- The full saved path (e.g.
  `.../20260607/SurveyedPerson_0001_Question_001_20260607_123456.webm`) is
  stored in `Response.audio_file_path_and_filename` as before.
- The function signature `save_audio_file(upload, surveyed_person_id, question_id)`
  is unchanged; callers in `routes/questionnaire.py` need no modification.

#### 8.1.3 Admin playback — canonical URL and dynamic MIME type

**Defect A — URL mismatch:** `admin/responses.html` builds modal audio sources
as:

```javascript
`/audio/${encodeURIComponent(item.audio_path)}`
```

The admin router is registered with `prefix="/admin"` and the audio route is
`@router.get("/audio/{file_path:path}")`, so the actual served URL is
`/admin/audio/{file_path}`. The template's `/audio/` prefix misses the
`/admin/` segment and hits no registered route, producing 404s for all audio
playback.

**Fix A:** Change the template's audio source URL to:

```javascript
`/admin/audio/${encodeURIComponent(item.audio_path)}`
```

No route change is needed; the route itself is already correctly placed on the
admin router. The path-traversal guard in the route handler (resolving the
joined path and asserting it is under `AUDIO_FILES_PATH`) is preserved intact.

**Defect B — hardcoded MIME type:** The route handler returns
`media_type="audio/mpeg"` for every file regardless of what is stored. After
§8.1.2 is applied, stored files will have `.webm` (and potentially other)
extensions. Serving WebM audio as `audio/mpeg` causes browsers to reject or
misrender the `<audio>` element.

**Fix B:** In the route handler, derive the MIME type from the stored file's
extension using Python's standard library:

```python
import mimetypes

media_type, _ = mimetypes.guess_type(str(target))
if not media_type or not media_type.startswith("audio/"):
    media_type = "audio/webm"
return FileResponse(str(target), media_type=media_type)
```

`mimetypes.guess_type` correctly maps `.webm` → `audio/webm`, `.mp3` →
`audio/mpeg`, `.ogg` → `audio/ogg`, etc. The explicit guard ensures a
non-audio MIME type (e.g. from a corrupt filename) is never served; the

## 10. Backup utility — admin-facing Postgres design

**The problem fixed in V02:** V01 said to adapt `db_backup.py` to
`pg_dump`-based dumps and left "exact CLI invocation/scheduling" as a TODO-phase
detail. Formy's backup utility is not just an operator-side script: the admin
dashboard exposes create, restore, list, download, and delete actions through
routes and UI — all of which the PRD requires to be preserved. The old
CSV-in-zip approach is SQLite-specific (the restore code includes a literal
`UPDATE sqlite_sequence` call) and cannot be carried forward.

### 10.1 Route surface preserved; handler and utility internals changed

The **HTTP route surface** — paths, methods, auth requirement, and JSON response
shapes — is preserved from Formy. The **handler internals** change materially
because the Postgres backup design does not use an ORM session for dump/restore
operations, does not accept `.zip` uploads, and serves files as binary downloads
rather than zip archives.

The table below documents exactly what is preserved and what changes for each
route:

| Method | Path | Preserved | Changed |
|--------|------|-----------|---------|
| `GET` | `/admin` | Path, auth (`require_admin`), dashboard renders backup list | Template shows `.dump` filenames and sizes |
| `POST` | `/admin/db/backup` | Path, method, auth, `{"ok": true, "filename": "..."}` JSON shape | Handler calls `create_backup()` with **no ORM session**; returns `.dump` filename |
| `POST` | `/admin/db/restore` | Path, method, auth requirement, error response shape | Handler uses **restore-specific auth boundary** (§10.4); validates `.dump` extension; calls `restore_backup(dump_bytes)` with **no ORM session open** |
| `GET` | `/admin/db/backups/{filename}` | Path, method, auth, path-traversal guard | Serves file with `media_type="application/octet-stream"` (not `application/zip`) |
| `DELETE` | `/admin/db/backups/{filename}` | Path, method, auth, `{"ok": true}` JSON shape | Globs `.dump` files, not `.zip` |

**Handler call signatures — explicitly different from Formy:**

Formy's `create_backup(db: Session)` and `restore_backup(db: Session, zip_bytes: bytes)`
accept an ORM session because the SQLite backup reads the database through SQLAlchemy.
The new utility functions read database connection parameters from config env vars
and shell out to `pg_dump`/`pg_restore`; no ORM session is involved:

- `POST /admin/db/backup` handler: calls `db_backup.create_backup()` — zero
  arguments; the function draws host/port/user/dbname from config. Returns
  `{"ok": True, "filename": dump_path.name}`.
- `POST /admin/db/restore` handler: uses a restore-specific auth dependency
  (§10.4) that guarantees no ORM session is open before the handler body runs;
  checks `backup_file.filename.endswith(".dump")` and returns HTTP 400 if not;
  reads the upload bytes; calls `db_backup.restore_backup(dump_bytes: bytes)` —
  one argument, no session.
- `GET /admin/db/backups/{filename}` handler: resolves the path through
  `get_backup_path(filename)` (path-traversal guard intact), then returns a
  `FileResponse` with `media_type="application/octet-stream"`.

Passing an ORM `Session` to these functions would be a bug; the utility
functions must not accept one.

### 10.2 Backup artifact format

Backups are produced by `pg_dump` in **custom format** (`-Fc`). Custom format
is binary, internally compressed, and directly consumed by `pg_restore` — it is
the appropriate choice for an in-app backup/restore cycle on a single-server
deployment.

Naming convention: `backup_YYYYMMDD_HHMMSS.dump` (`.dump` replaces Formy's
`.zip`). Files are stored in `{PATH_PROJECT_RESOURCES}/backups_db/`, the same
directory Formy uses, created with `mkdir -p` on first use.

`list_backups()` globs `backup_*.dump` and returns `{name, size_kb, modified}`
dicts exactly as before; the dashboard table is unchanged.

### 10.3 create_backup() implementation

`db_backup.create_backup()` invokes `pg_dump` via `subprocess.run` with an
explicit argument list (no `shell=True`). The DB connection parameters — host,
port, database name, and user — are drawn from the same config env vars used by
the ORM. `PGPASSWORD` is not set because Formy02 uses passwordless localhost
trust auth.

```
pg_dump \
  --format=custom \
  --no-owner \
  --no-privileges \
  --file={backups_dir}/backup_{timestamp}.dump \
  --host={DB_HOST} --port={DB_PORT} \
  --username={DB_USER} {DB_NAME}
```

The function raises `RuntimeError` if the subprocess exits non-zero; the route
catches `RuntimeError` and returns HTTP 500, consistent with Formy's existing
pattern.

### 10.4 restore_backup() implementation — session boundary and timeout

#### 10.4.1 The deadlock risk from the standard auth dependency

The standard `require_admin` dependency in Formy's auth design calls
`get_current_user`, which in turn depends on `get_session`. FastAPI resolves
`get_session` via `yield`, opening a SQLAlchemy/SQLModel session at the start
of the request and holding it open until after the response is sent and the
generator's cleanup code runs. SQLAlchemy's default connection-level behavior
begins an implicit transaction with the first query issued on that session —
in this case, the `SELECT ... FROM users WHERE ...` performed to validate the
admin cookie. That transaction is not committed or rolled back until the session
is explicitly closed or the `yield`-based dependency's cleanup runs.

When `pg_restore --clean` starts, it connects to Postgres under its own
connection and issues `DROP TABLE users` (among other DDL statements). A `DROP
TABLE` requires an exclusive lock on the target table. PostgreSQL cannot grant
that lock while the app's ORM connection holds an uncommitted transaction that
touched `users`. The `pg_restore` subprocess therefore blocks waiting for the
app's ORM session to release its lock. But the app's ORM session does not
release its lock until the request handler returns — which cannot happen until
the subprocess returns. This is a deadlock that will hang the restore request
indefinitely.

#### 10.4.2 Why pg_terminate_backend is not sufficient alone

V07 noted that a `SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE
datname = $1 AND pid <> pg_backend_pid()` step could be added before or during
restore. This is not sufficient to break the deadlock described above because:

- The connection holding the blocking lock is the **current request's own ORM
  connection** — the one opened by the `get_session` dependency for the restore
  request itself.
- `pg_terminate_backend` terminates *other* connections. It cannot terminate
  the connection it is being called on, and even if it could, doing so would
  raise an error in the same request handler that is trying to run the restore.
- If `pg_terminate_backend` is called from within the restore handler (before
  shelling out to `pg_restore`), it would terminate any other app connections
  active at that moment, but the auth dependency's ORM session on the current
  request remains open and continues to hold its lock.
- In short: `pg_terminate_backend` clears *other* connections from the pool but
  does nothing about the lock held by the current request's own session. The
  deadlock persists.

The correct fix is to ensure the current request's ORM session is closed
**before** the `pg_restore` subprocess starts, not to terminate other
connections after the fact.

#### 10.4.3 Restore-specific auth dependency

The `POST /admin/db/restore` route must use a **restore-specific auth
dependency** — `require_admin_for_restore` — instead of the standard
`require_admin`. The purpose is to perform admin authentication through a
short-lived, explicitly closed session that leaves no open transaction by the
time the handler body executes.

Canonical design:

```python
def require_admin_for_restore(
    request: Request,
    engine: Engine = Depends(get_engine),
) -> User:
    """Auth dependency for the restore route.

    Opens a session, validates the signed cookie and admin flag, then
    explicitly closes the session before returning. The caller's handler body
    runs with no ORM connection open, so pg_restore can acquire the exclusive
    locks it needs for --clean without deadlocking.
    """
    with Session(engine) as session:
        user = _validate_admin_cookie(request, session)
    # Session is committed/rolled back and the connection returned to the pool
    # here, before the handler body runs. No open transaction remains.
    return user
```

`_validate_admin_cookie` is the shared helper extracted from the existing
`get_current_user` logic: it reads the signed session cookie or JWT, queries the
`users` table, checks `email_verified` and `is_admin`, and raises
`HTTPException(401)` or `HTTPException(403)` on failure. The `with Session(...)`
context manager rolls back any implicit transaction and returns the connection to
the pool when the block exits, regardless of success or failure.

The restore handler signature becomes:

```python
@router.post("/db/restore")
async def restore_database(
    backup_file: UploadFile,
    admin: User = Depends(require_admin_for_restore),  # session closed on return
):
    if not backup_file.filename.endswith(".dump"):
        raise HTTPException(status_code=400, detail="File must be a .dump archive")
    dump_bytes = await backup_file.read()
    db_backup.restore_backup(dump_bytes)  # no ORM session open at this point
    return {"ok": True}
```

`require_admin_for_restore` must not accept a `Session` as a parameter and must
not yield — it must open and close the session entirely within its own scope
before returning, guaranteeing the dependency-injection cleanup path cannot keep
a session alive past its return.

#### 10.4.4 restore_backup() subprocess with bounded timeout

`restore_backup(dump_bytes: bytes)` writes the bytes to a named temporary file
(via `tempfile`), then runs `pg_restore` via `subprocess.run` with an explicit
argument list:

```
pg_restore \
  --clean \
  --if-exists \
  --no-owner \
  --no-privileges \
  --format=custom \
  --host={DB_HOST} --port={DB_PORT} \
  --username={DB_USER} \
  --dbname={DB_NAME} \
  {tempfile_path}
```

The subprocess call must include a hard timeout. The recommended ceiling is 300
seconds (5 minutes), which is generous for any realistic database size at this
deployment scale:

```python
try:
    subprocess.run(
        [...],
        env={**os.environ, "PGOPTIONS": "-c lock_timeout=30s"},
        check=True,
        timeout=300,
    )
except subprocess.TimeoutExpired as exc:
    raise RuntimeError(
        "pg_restore timed out after 300 seconds; restore did not complete"
    ) from exc
```

Set `PGOPTIONS="-c lock_timeout=30s"` in the subprocess environment. This instructs Postgres to abort any individual lock-wait
within `pg_restore` that exceeds 30 seconds rather than blocking forever. If lock
contention arises from a connection that was not visible at the time of the auth
check (e.g. a background worker that connected after the auth session closed),
the Postgres-level timeout prevents that contention from becoming a permanent
hang. The two timeouts are complementary: `lock_timeout` aborts early on
unexpected lock waits; `subprocess.run(timeout=...)` is the process-level ceiling
that guards against any other cause of indefinite blocking.

The temporary file is deleted in a `finally` block regardless of success,
timeout, or error:

```python
with tempfile.NamedTemporaryFile(suffix=".dump", delete=False) as tmp:
    tmp.write(dump_bytes)
    tmp_path = tmp.name
try:
    subprocess.run([...], timeout=300, check=True, env=restore_env)
except subprocess.TimeoutExpired as exc:
    raise RuntimeError("pg_restore timed out") from exc
finally:
    Path(tmp_path).unlink(missing_ok=True)
```

The route catches `RuntimeError` and returns HTTP 500, consistent with the
backup route's error handling pattern.

## 11. GoLightly04-side change (separate repo)

Reference: `GoLightly04/web/src/components/AppShell.tsx` (footer "Help Us
Improve" anchor), `ThemeToggle.tsx` (`golightly.theme` in `localStorage`,
`.dark` on `<html>`).

### 11.1 Goal and constraint

The footer anchor currently renders as:

```tsx
<a
  href="https://formy.go-lightly.love"
  target="_blank"
  rel="noreferrer"
>
  Help Us Improve
</a>
```

The sole goal is to make the link forward the active GoLightly04 theme to
Formy02 via the `?theme=` query parameter. **The existing anchor semantics —
opening in a new tab (`target="_blank"`) with `rel="noreferrer"` — must be
preserved exactly.** The PRD does not ask to change how the link opens.

### 11.2 Approved implementation approaches

Two approaches are acceptable; both keep the `<a>` tag with its existing
`target` and `rel` attributes:

**Approach A — computed href (recommended):** Derive the current theme from
React state (reading `localStorage.getItem('golightly.theme')` or observing
`document.documentElement.classList.contains('dark')`) and render a dynamic
`href`:

```tsx
const currentTheme = /* read from state or localStorage */ 'light' | 'dark';
<a
  href={`https://formy.go-lightly.love?theme=${currentTheme}`}
  target="_blank"
  rel="noreferrer"
>
  Help Us Improve
</a>
```

The browser's default anchor behavior opens the computed URL in a new tab.
No `onClick` handler is needed.

**Approach B — onClick href mutation:** Attach an `onClick` handler that
updates `event.currentTarget.href` to include the current theme immediately
before the browser follows the link. The handler must **not** call
`event.preventDefault()` and must **not** assign to `window.location` or
`location.href`:

```tsx
const handleClick = (event: React.MouseEvent<HTMLAnchorElement>) => {
  const theme = localStorage.getItem('golightly.theme') === 'dark' ? 'dark' : 'light';
  event.currentTarget.href = `https://formy.go-lightly.love?theme=${theme}`;
  // No preventDefault — browser default behavior (target="_blank") proceeds
};

<a
  href="https://formy.go-lightly.love"
  target="_blank"
  rel="noreferrer"
  onClick={handleClick}
>
  Help Us Improve
</a>
```

### 11.3 Prohibited patterns

The following patterns must not be used, as they navigate the current tab and
drop the new-tab / noreferrer semantics:

- `window.location.href = ...`
- `location.href = ...`
- `event.preventDefault()` followed by programmatic navigation (unless the
  handler also explicitly calls `window.open(..., '_blank', 'noreferrer')`,
  which is more complex and unnecessary when Approach A or B is available)
- Converting the `<a>` element to a `<button>` or `<div>` with an `onClick`
  that performs navigation

### 11.4 Isolation and sequencing

No other GoLightly changes are in scope. This change is isolated and low risk;
it is testable independently by verifying that clicking the footer link opens a
new tab with the correct `?theme=` value. Implement as the final step so
Formy02's `?theme=` handling already exists to receive it.
